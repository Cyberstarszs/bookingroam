CATATAN PENYESUAIAN DOKUMEN
================================================================================

Dokumen SRS ini telah disesuaikan dari versi lengkap (enterprise-grade) menjadi
versi yang realistis untuk dikerjakan oleh 2 orang dalam waktu terbatas (proyek
UAS), dengan tetap memenuhi seluruh materi yang wajib dibuktikan sesuai panduan
UAS Pemrograman Web. Penyesuaian yang dilakukan:

  1. Peran pengguna disederhanakan menjadi 2 peran saja: ADMIN dan USER
     (peran "Staff" pada versi lengkap dihapus untuk versi UAS ini)
  2. Fitur-fitur berlabel [OPSIONAL] pada versi lengkap sebagian besar TIDAK
     dikerjakan, kecuali jika waktu memungkinkan
  3. Tidak ada integrasi pihak ketiga yang rumit (SSO, WhatsApp Gateway, OTP SMS)
  4. JWT disimpan di Authorization Header (Bearer Token), bukan HTTP-only Cookie
  5. Notifikasi disederhanakan menjadi notifikasi IN-APP saja (tanpa email SMTP),
     untuk memangkas kompleksitas infrastruktur yang tidak relevan dengan
     penilaian UAS
  6. Struktur backend WAJIB mengikuti arsitektur Router - Controller - Model/
     Service sesuai tuntutan rubrik penilaian
  7. Setiap modul fungsional dipetakan secara eksplisit ke materi kuliah
     (lihat Bab 3.0 - Pemetaan Modul ke Materi Kuliah)

Dokumen ini dirancang agar 2 anggota kelompok dapat membagi tugas secara adil
dan jelas, sesuai tuntutan penilaian "Kontribusi Git & Kolaborasi" (10%).

BAB 1 — PENDAHULUAN
================================================================================

1.1 TUJUAN DOKUMEN
--------------------------------------------------------------------------------
Dokumen ini menjelaskan kebutuhan fungsional dan non-fungsional dari Sistem
Reservasi Ruangan yang dikembangkan sebagai proyek UAS mata kuliah Pemrograman
Web. Dokumen ini berfungsi sebagai:

  - Acuan teknis pengembangan untuk 2 anggota kelompok
  - Dasar pembagian tugas yang adil dan terukur
  - Checklist pemenuhan materi kuliah (Minggu 2-14) yang wajib dibuktikan
  - Bahan persiapan untuk sesi Oral Defense dan Live Modification saat sidang


1.2 RUANG LINGKUP PROYEK
--------------------------------------------------------------------------------
Sistem Reservasi Ruangan versi UAS ini mencakup:

  a) Registrasi dan login pengguna dengan password hashing (bcrypt)
  b) Autentikasi berbasis JWT (Bearer Token via Authorization Header)
  c) Dua peran pengguna: ADMIN dan USER
  d) CRUD data ruangan (oleh Admin)
  e) Proses pengajuan, persetujuan, dan pembatalan reservasi ruangan
  f) Notifikasi in-app sederhana (tanpa email)
  g) Dashboard ringkas sesuai peran
  h) Frontend SPA menggunakan Svelte dengan client-side routing
  i) Backend RESTful API menggunakan Express.js dengan arsitektur berlapis
  j) Database relasional MariaDB dengan Drizzle ORM

Yang TIDAK termasuk dalam ruang lingkup versi UAS ini:
  - Pembayaran/transaksi keuangan
  - Notifikasi email/SMTP, WhatsApp, atau SMS
  - Single Sign-On (SSO) atau OTP dua faktor
  - Reservasi berulang (recurring booking)
  - Fitur check-in/check-out fisik
  - Laporan analitik lanjutan (heatmap, prediksi, dsb.)
  - Peran "Staff" terpisah dari Admin


1.3 DEFINISI, AKRONIM, DAN SINGKATAN
--------------------------------------------------------------------------------
Istilah          Definisi
-----------      ---------------------------------------------------------------
SRS              Software Requirements Specification
SPA              Single Page Application
JWT              JSON Web Token, digunakan untuk autentikasi stateless
Bearer Token     Skema autentikasi HTTP, token dikirim di header:
                 Authorization: Bearer <token>
bcrypt           Algoritma hashing satu arah untuk mengamankan password
ORM              Object Relational Mapping (di proyek ini: Drizzle ORM)
CRUD             Create, Read, Update, Delete
CORS             Cross-Origin Resource Sharing
Reservasi        Proses pengajuan pemakaian ruangan pada waktu tertentu
Admin            Pengguna dengan akses penuh: kelola ruangan, setujui reservasi
User             Pengguna umum yang mengajukan reservasi ruangan
Middleware       Fungsi perantara di Express.js yang dijalankan sebelum handler
                 utama (digunakan untuk autentikasi & otorisasi)
Endpoint         Alamat URL API yang dapat diakses oleh frontend

BAB 2 — DESKRIPSI UMUM SISTEM
================================================================================

2.1 PERSPEKTIF PRODUK
--------------------------------------------------------------------------------
Sistem berupa aplikasi web fullstack dengan pemisahan jelas antara:

  - Frontend: Svelte SPA (berjalan di browser, melakukan fetch ke backend)
  - Backend: Express.js RESTful API (menangani logika bisnis dan akses database)
  - Database: MariaDB (diakses backend melalui Drizzle ORM)

Frontend dan backend berjalan sebagai dua proyek terpisah (dua port berbeda saat
development), dihubungkan melalui HTTP request dengan format JSON, dan
dikonfigurasi CORS agar frontend dapat mengakses backend dengan aman.


2.2 FUNGSI UTAMA PRODUK
--------------------------------------------------------------------------------
  F-01  Registrasi dan login pengguna (dengan password hashing)
  F-02  Autentikasi dan otorisasi berbasis JWT dan role (Admin/User)
  F-03  Pengelolaan data ruangan (tambah, lihat, ubah, hapus) — khusus Admin
  F-04  Pencarian ruangan tersedia berdasarkan tanggal dan jam
  F-05  Pengajuan reservasi ruangan oleh User
  F-06  Persetujuan/penolakan reservasi oleh Admin
  F-07  Pembatalan reservasi oleh User (pemilik) atau Admin
  F-08  Notifikasi in-app atas perubahan status reservasi
  F-09  Dashboard ringkas berbeda untuk Admin dan User


2.3 KARAKTERISTIK PENGGUNA (2 PERAN)
--------------------------------------------------------------------------------

[A] ADMIN
    Deskripsi     : Pengelola sistem, biasanya petugas/pengelola gedung
    Jumlah akun   : Minimal 1 akun (dibuat lewat seeder database)
    Akses         :
      - Mengelola seluruh data ruangan (CRUD)
      - Melihat seluruh reservasi dari semua user
      - Menyetujui atau menolak reservasi yang berstatus "pending"
      - Membatalkan reservasi siapapun jika diperlukan
      - Melihat dashboard ringkas sistem

[B] USER
    Deskripsi     : Pengguna umum (mahasiswa, dosen, staf, dll.)
    Jumlah akun   : Tidak terbatas, dibuat melalui registrasi mandiri
    Akses         :
      - Mendaftar dan login ke sistem
      - Mencari ruangan yang tersedia
      - Mengajukan reservasi ruangan
      - Melihat riwayat dan status reservasi miliknya sendiri
      - Membatalkan reservasi miliknya sendiri (jika masih "pending")
      - Melihat notifikasi terkait reservasinya


2.4 ASUMSI DAN BATASAN PROYEK
--------------------------------------------------------------------------------
  a) Proyek dikerjakan oleh 2 orang dalam jangka waktu satu periode UAS
  b) Akun Admin pertama dibuat melalui seed data, tidak ada fitur "daftar
     sebagai admin" melalui form registrasi publik
  c) Tidak ada pembatasan jumlah ruangan atau jumlah reservasi yang signifikan
     mengingat skala proyek kecil (cukup untuk demo)
  d) Database yang digunakan adalah MariaDB, diakses melalui Drizzle ORM
  e) Validasi data dilakukan baik di sisi frontend (UX) maupun backend (wajib,
     untuk keamanan)
  f) Karena waktu terbatas, dokumentasi README dan video demo harus disiapkan
     sejak awal pengerjaan, bukan di akhir

4.1 SKEMA PEMBAGIAN TUGAS
--------------------------------------------------------------------------------
Karena kelompok hanya 2 orang, pembagian dilakukan berdasarkan domain
(bukan murni frontend vs backend) agar masing-masing anggota memahami penuh
satu alur fitur dari ujung ke ujung — ini PENTING untuk Oral Defense, karena
penguji akan menanyakan baris kode spesifik milik masing-masing orang.

ANGGOTA A — "Pemilik Modul Autentikasi & Manajemen Ruangan"
  Backend:
    - Setup project Express.js, struktur folder, koneksi Drizzle ORM ke MariaDB
    - Skema tabel: users, rooms
    - Endpoint: register, login, CRUD ruangan
    - Middleware: verifyToken, verifyAdmin
    - Implementasi bcrypt untuk hashing password
    - Implementasi JWT generation & verification
  Frontend:
    - Halaman Login & Register
    - Halaman Daftar Ruangan (grid CSS Grid) & Detail Ruangan
    - Halaman Manajemen Ruangan (Admin): tambah/edit/hapus ruangan
    - Svelte Store untuk menyimpan data user yang login (authStore)

ANGGOTA B — "Pemilik Modul Reservasi & Notifikasi"
  Backend:
    - Skema tabel: reservations, notifications
    - Endpoint: ajukan reservasi, lihat reservasi, setujui/tolak, batalkan
    - Logika validasi bisnis: cek bentrok jadwal, validasi waktu
    - Endpoint notifikasi in-app
  Frontend:
    - Form Pengajuan Reservasi
    - Halaman Riwayat Reservasi (User) & Manajemen Reservasi (Admin)
    - Dashboard (ringkasan berbeda Admin/User)
    - Komponen Notifikasi (lonceng + daftar notifikasi)

TUGAS BERSAMA (dikerjakan berdua / pair programming):
    - Konfigurasi CORS dan integrasi end-to-end frontend-backend
    - Routing SPA (client-side routing) keseluruhan aplikasi
    - Styling global, animasi/transisi CSS, responsivitas
    - Penulisan README.md, dokumentasi API, dan .env.example
    - Rekaman video demo


4.2 RENCANA COMMIT GIT
--------------------------------------------------------------------------------
Sesuai kebijakan anti-vibecoding, minimal 5 commit bermakna per anggota,
bertahap (bukan bulk commit di akhir). Contoh rencana commit:

  2. "feat: implement users schema & register/login endpoint"

ANGGOTA B:

BAB 5 — KEBUTUHAN FUNGSIONAL
================================================================================

Notasi: FR-XX = Functional Requirement, [WAJIB] = harus ada untuk lulus rubrik
penilaian "Fungsionalitas Fullstack & DB" (20%)

--------------------------------------------------------------------------------
5.1 MODUL AUTENTIKASI & OTORISASI
--------------------------------------------------------------------------------
FR-01 [WAJIB] Registrasi Akun (User)
      - Form input: nama, email, password, konfirmasi password
      - Validasi: email unik, password minimal 6 karakter
      - Password di-hash dengan bcrypt sebelum disimpan ke tabel users
      - Role default saat registrasi mandiri adalah "user"

FR-02 [WAJIB] Login
      - Form input: email, password
      - Backend memverifikasi password dengan bcrypt.compare()
      - Jika valid, backend membuat JWT berisi payload {id, role}
      - Token dikembalikan ke frontend dan disimpan di localStorage
      - Frontend menyertakan token di header Authorization pada setiap
        request ke endpoint yang terproteksi

FR-03 [WAJIB] Middleware Autentikasi (verifyToken)
      - Membaca header Authorization: Bearer <token>
      - Memverifikasi keabsahan token (jwt.verify)
      - Jika tidak valid/tidak ada, kembalikan 401 Unauthorized
      - Jika valid, lanjutkan ke handler dan sisipkan data user di req.user

FR-04 [WAJIB] Middleware Otorisasi (verifyAdmin)
      - Mengecek req.user.role setelah verifyToken
      - Jika role bukan "admin", kembalikan 403 Forbidden
      - Diterapkan pada endpoint khusus admin (kelola ruangan, approve
        reservasi siapapun)

FR-05 [WAJIB] Logout
      - Frontend menghapus token dari localStorage
      - Redirect ke halaman login


--------------------------------------------------------------------------------
5.2 MODUL DASHBOARD
--------------------------------------------------------------------------------
FR-06 [WAJIB] Dashboard Admin
      - Jumlah total ruangan
      - Jumlah reservasi berstatus "pending" (perlu ditindak)
      - Daftar ringkas reservasi terbaru

FR-07 [WAJIB] Dashboard User
      - Daftar reservasi milik sendiri yang akan datang
      - Status reservasi terakhir yang diajukan
      - Tombol cepat "Ajukan Reservasi Baru"


--------------------------------------------------------------------------------
5.3 MODUL MANAJEMEN RUANGAN (Khusus Admin)
--------------------------------------------------------------------------------
FR-08 [WAJIB] Tambah Ruangan
      Field: nama ruangan, lokasi, kapasitas, deskripsi, jam_buka, jam_tutup
      Endpoint terproteksi middleware verifyAdmin

FR-09 [WAJIB] Lihat Daftar Ruangan
      - Dapat diakses oleh Admin dan User (publik setelah login)
      - Ditampilkan dalam grid (CSS Grid) berisi nama, kapasitas, status

FR-10 [WAJIB] Edit Ruangan
      Admin dapat mengubah seluruh data ruangan

FR-11 [WAJIB] Hapus Ruangan
      - Hanya dapat dilakukan jika tidak ada reservasi aktif terkait ruangan
      - Tampilkan dialog konfirmasi sebelum hapus


--------------------------------------------------------------------------------
5.4 MODUL RESERVASI
--------------------------------------------------------------------------------
FR-12 [WAJIB] Ajukan Reservasi (User)
      Field: room_id, tanggal, jam_mulai, jam_selesai, keperluan
      Validasi backend:
        - Ruangan tidak sedang dipesan pada slot waktu yang sama
          (cek overlap waktu dengan query database)
        - jam_mulai < jam_selesai
        - tanggal tidak boleh di masa lampau
      Status awal: "pending"

FR-13 [WAJIB] Lihat Reservasi Milik Sendiri (User)
      - User hanya dapat melihat reservasi miliknya (filter berdasarkan
        req.user.id dari token, bukan dari input client)

FR-14 [WAJIB] Lihat Semua Reservasi (Admin)
      - Admin dapat melihat seluruh reservasi dari semua user
      - Filter berdasarkan status: pending / approved / rejected / cancelled

FR-15 [WAJIB] Setujui / Tolak Reservasi (Admin)
      - Admin dapat mengubah status "pending" menjadi "approved" atau
        "rejected"
      - Saat menolak, admin mengisi alasan penolakan (field opsional)
      - Sistem membuat notifikasi in-app untuk user pemilik reservasi

FR-16 [WAJIB] Batalkan Reservasi
      - User dapat membatalkan reservasi miliknya sendiri jika statusnya
        masih "pending" atau "approved"
      - Admin dapat membatalkan reservasi siapapun
      - Status berubah menjadi "cancelled"


--------------------------------------------------------------------------------
5.5 MODUL NOTIFIKASI (IN-APP)
--------------------------------------------------------------------------------
FR-17 [WAJIB] Buat Notifikasi Otomatis
      Sistem membuat baris baru di tabel notifications saat:
        - Reservasi disetujui → notifikasi ke user
        - Reservasi ditolak → notifikasi ke user (beserta alasan)
        - Reservasi baru diajukan → notifikasi ke semua admin (opsional,
          bisa disederhanakan jadi badge counter di dashboard admin)

FR-18 [WAJIB] Lihat Notifikasi
      - Ikon lonceng di navbar menampilkan jumlah notifikasi belum dibaca
      - Klik menampilkan daftar notifikasi terbaru milik user yang login


--------------------------------------------------------------------------------
5.6 MODUL MANAJEMEN PENGGUNA (Disederhanakan)
--------------------------------------------------------------------------------
FR-19 [OPSIONAL] Admin melihat daftar user terdaftar
      Cukup tabel sederhana: nama, email, tanggal daftar. Tidak perlu fitur
      edit/hapus user kompleks untuk versi UAS — fokuskan waktu ke modul
      reservasi yang menjadi inti penilaian.

BAB 6 — KEBUTUHAN NON-FUNGSIONAL
================================================================================

NFR-01  Password tidak boleh disimpan dalam bentuk plain text di database
        (wajib di-hash dengan bcrypt, minimal 10 salt rounds)
NFR-02  Token JWT memiliki masa berlaku (expiry), disarankan 24 jam untuk
        kemudahan testing/demo
NFR-03  Seluruh endpoint yang mengubah data (POST/PUT/DELETE) pada resource
        sensitif harus melalui middleware verifyToken
NFR-04  Validasi input dilakukan di backend (tidak hanya mengandalkan validasi
        HTML/frontend), karena ini yang dinilai saat live modification

7.4 ALUR AUTENTIKASI JWT (BEARER TOKEN)
--------------------------------------------------------------------------------
  1. User login → backend verifikasi password → backend generate JWT
     (jwt.sign({id, role}, SECRET, {expiresIn: '24h'}))
  2. Backend mengembalikan { token, user: {id, name, email, role} }
  3. Frontend menyimpan token di localStorage
  4. Setiap request ke endpoint terproteksi, frontend menambahkan header:
        Authorization: Bearer <token>
  5. Middleware verifyToken di backend membaca header, memverifikasi dengan
     jwt.verify(), lalu menyisipkan payload ke req.user
  6. Jika endpoint khusus admin, middleware verifyAdmin mengecek req.user.role
  7. Jika token tidak ada/invalid/expired → backend mengembalikan 401
  8. Jika role tidak sesuai → backend mengembalikan 403
  9. Frontend menangkap status 401 secara global (di wrapper api.js) dan
     otomatis redirect ke halaman login jika sesi tidak valid


7.5 PENANGANAN CORS
--------------------------------------------------------------------------------
  - Backend mengaktifkan middleware cors() dari package "cors"
  - Konfigurasi origin diarahkan secara spesifik ke alamat frontend
    (contoh: origin: 'http://localhost:5173'), TIDAK menggunakan origin: '*'
    pada endpoint yang memerlukan kredensial
  - Method yang diizinkan: GET, POST, PUT, DELETE
  - Header yang diizinkan: Content-Type, Authorization

8.1 ENTITY RELATIONSHIP DIAGRAM (ERD - TEKS)
--------------------------------------------------------------------------------

  USERS (1) ────────< (N) RESERVATIONS
  ROOMS (1) ────────< (N) RESERVATIONS

    - Satu ROOM dapat memiliki banyak RESERVATIONS  (One-to-Many)
    - Satu USER dapat memiliki banyak NOTIFICATIONS  (One-to-Many)


8.2 SKEMA TABEL LENGKAP (UNTUK DRIZZLE ORM)
--------------------------------------------------------------------------------

[TABEL: users]
  id            INT, PK, AUTO_INCREMENT
  name          VARCHAR(100), NOT NULL
  email         VARCHAR(150), UNIQUE, NOT NULL
  password      VARCHAR(255), NOT NULL        -- hasil bcrypt hash
  role          ENUM('admin', 'user'), DEFAULT 'user'
  created_at    TIMESTAMP, DEFAULT CURRENT_TIMESTAMP

[TABEL: rooms]
  id            INT, PK, AUTO_INCREMENT
  name          VARCHAR(150), NOT NULL, UNIQUE
  location      VARCHAR(200)
  capacity      INT, NOT NULL
  description   TEXT
  open_time     TIME, NOT NULL, DEFAULT '08:00:00'
  close_time    TIME, NOT NULL, DEFAULT '17:00:00'
  created_at    TIMESTAMP, DEFAULT CURRENT_TIMESTAMP

[TABEL: reservations]
  id            INT, PK, AUTO_INCREMENT
  user_id       INT, FK → users.id, NOT NULL
  room_id       INT, FK → rooms.id, NOT NULL
  date          DATE, NOT NULL
  start_time    TIME, NOT NULL
  end_time      TIME, NOT NULL
  purpose       VARCHAR(255), NOT NULL        -- keperluan reservasi
  status        ENUM('pending','approved','rejected','cancelled'),
                DEFAULT 'pending'
  reject_reason VARCHAR(255), NULL
  created_at    TIMESTAMP, DEFAULT CURRENT_TIMESTAMP

[TABEL: notifications]
  id            INT, PK, AUTO_INCREMENT
  user_id       INT, FK → users.id, NOT NULL
  message       VARCHAR(255), NOT NULL
  is_read       BOOLEAN, DEFAULT false
  created_at    TIMESTAMP, DEFAULT CURRENT_TIMESTAMP

Contoh definisi skema Drizzle ORM (gambaran, sesuaikan dengan dialect MariaDB):

  export const users = mysqlTable('users', {
    id: int('id').autoincrement().primaryKey(),
    name: varchar('name', { length: 100 }).notNull(),
    email: varchar('email', { length: 150 }).notNull().unique(),
    password: varchar('password', { length: 255 }).notNull(),
    role: mysqlEnum('role', ['admin', 'user']).default('user'),
    createdAt: timestamp('created_at').defaultNow(),
  });

  export const reservations = mysqlTable('reservations', {
    id: int('id').autoincrement().primaryKey(),
    userId: int('user_id').notNull().references(() => users.id),
    roomId: int('room_id').notNull().references(() => rooms.id),
    date: date('date').notNull(),
    startTime: time('start_time').notNull(),
    endTime: time('end_time').notNull(),
    purpose: varchar('purpose', { length: 255 }).notNull(),
    status: mysqlEnum('status',
      ['pending','approved','rejected','cancelled']).default('pending'),
    rejectReason: varchar('reject_reason', { length: 255 }),
    createdAt: timestamp('created_at').defaultNow(),
  });

  -- Query pengecekan bentrok jadwal (logika inti FR-12):
  -- Cari reservasi di room yang sama, tanggal sama, status bukan
  -- 'rejected'/'cancelled', dan rentang waktu saling overlap:
  --   (start_time < :end_time) AND (end_time > :start_time)

BAB 9 — DOKUMENTASI API ENDPOINT
================================================================================

Catatan format: Body = payload request, Auth = perlu header Authorization,
Role = peran yang diizinkan mengakses endpoint.

--------------------------------------------------------------------------------
9.1 ENDPOINT AUTENTIKASI
--------------------------------------------------------------------------------

POST   /api/auth/register
  Auth   : Tidak perlu
  Body   : { "name": string, "email": string, "password": string }
  Respon : 201 { "message": "Registrasi berhasil", "user": {...} }

POST   /api/auth/login
  Auth   : Tidak perlu
  Body   : { "email": string, "password": string }
  Respon : 200 { "token": "<jwt>", "user": { "id", "name", "email", "role" } }
  Error  : 401 { "message": "Email atau password salah" }

GET    /api/auth/me
  Auth   : Bearer Token
  Respon : 200 { "id", "name", "email", "role" }
  Error  : 401 { "message": "Token tidak valid" }


--------------------------------------------------------------------------------
9.2 ENDPOINT RUANGAN
--------------------------------------------------------------------------------

GET    /api/rooms
  Auth   : Bearer Token (semua role)
  Query  : ?date=YYYY-MM-DD&start=HH:mm&end=HH:mm  (opsional, untuk filter
           ruangan tersedia pada slot waktu tertentu)
  Respon : 200 [ { "id", "name", "location", "capacity", ... } ]

GET    /api/rooms/:id
  Auth   : Bearer Token
  Respon : 200 { "id", "name", "location", "capacity", "description", ... }
  Error  : 404 { "message": "Ruangan tidak ditemukan" }

POST   /api/rooms
  Auth   : Bearer Token, Role: admin
  Body   : { "name", "location", "capacity", "description",
             "open_time", "close_time" }
  Respon : 201 { "message": "Ruangan berhasil ditambahkan", "room": {...} }

PUT    /api/rooms/:id
  Auth   : Bearer Token, Role: admin
  Body   : { field yang ingin diubah }
  Respon : 200 { "message": "Ruangan berhasil diperbarui" }

DELETE /api/rooms/:id
  Auth   : Bearer Token, Role: admin
  Respon : 200 { "message": "Ruangan berhasil dihapus" }
  Error  : 400 { "message": "Ruangan masih memiliki reservasi aktif" }


--------------------------------------------------------------------------------
9.3 ENDPOINT RESERVASI
--------------------------------------------------------------------------------

POST   /api/reservations
  Auth   : Bearer Token (role: user/admin)
  Body   : { "room_id", "date", "start_time", "end_time", "purpose" }
  Respon : 201 { "message": "Reservasi berhasil diajukan", "reservation": {...} }
  Error  : 409 { "message": "Ruangan sudah dipesan pada waktu tersebut" }
           400 { "message": "Waktu tidak valid" }

GET    /api/reservations/me
  Auth   : Bearer Token
  Respon : 200 [ { "id", "room", "date", "start_time", "end_time", "status" } ]
  Catatan: backend WAJIB memfilter berdasarkan req.user.id dari token, bukan
           dari parameter yang dikirim client (mencegah user melihat data
           milik orang lain)

GET    /api/reservations
  Auth   : Bearer Token, Role: admin
  Query  : ?status=pending|approved|rejected|cancelled
  Respon : 200 [ { "id", "user", "room", "date", "status", ... } ]

PATCH  /api/reservations/:id/approve
  Auth   : Bearer Token, Role: admin
  Respon : 200 { "message": "Reservasi disetujui" }

PATCH  /api/reservations/:id/reject
  Auth   : Bearer Token, Role: admin
  Body   : { "reject_reason": string }
  Respon : 200 { "message": "Reservasi ditolak" }

PATCH  /api/reservations/:id/cancel
  Auth   : Bearer Token (pemilik reservasi atau admin)
  Respon : 200 { "message": "Reservasi dibatalkan" }
  Error  : 403 { "message": "Tidak diizinkan membatalkan reservasi ini" }


--------------------------------------------------------------------------------
9.4 ENDPOINT PENGGUNA
--------------------------------------------------------------------------------

GET    /api/users
  Auth   : Bearer Token, Role: admin
  Respon : 200 [ { "id", "name", "email", "role", "created_at" } ]
  (Opsional, lihat FR-19)


--------------------------------------------------------------------------------
9.5 ENDPOINT NOTIFIKASI
--------------------------------------------------------------------------------

GET    /api/notifications
  Auth   : Bearer Token
  Respon : 200 [ { "id", "message", "is_read", "created_at" } ]
  Catatan: hanya mengembalikan notifikasi milik req.user.id

PATCH  /api/notifications/:id/read
  Auth   : Bearer Token
  Respon : 200 { "message": "Notifikasi ditandai sudah dibaca" }


--------------------------------------------------------------------------------
9.6 FORMAT RESPONS STANDAR & ERROR HANDLING
--------------------------------------------------------------------------------
Respon sukses:
  { "success": true, "message": "...", "data": { ... } }

Respon error:
  { "success": false, "message": "..." }

Kode status HTTP yang digunakan secara konsisten:
  200  OK (berhasil, GET/PUT/PATCH)
  201  Created (berhasil, POST data baru)
  400  Bad Request (validasi gagal)
  401  Unauthorized (token tidak ada/invalid)
  403  Forbidden (role tidak diizinkan)
  404  Not Found (data tidak ditemukan)
  409  Conflict (jadwal bentrok)
  500  Internal Server Error

Catatan untuk Live Modification: penguji kemungkinan akan meminta perubahan
format respons error (misal menambah field "errorCode"), jadi pastikan logika
error handling terpusat di satu tempat (middleware error handler), bukan
ditulis berulang di setiap controller, supaya mudah diubah saat sidang.

BAB 13 — SKENARIO PENGGUNAAN (USE CASE)
================================================================================

UC-01  MENGAJUKAN RESERVASI
  Aktor       : User
  Alur Normal :
    1. User login, lalu membuka halaman "Daftar Ruangan"
    2. User memilih ruangan, klik "Ajukan Reservasi"
    3. User mengisi tanggal, jam mulai, jam selesai, dan keperluan
    4. Sistem mengirim POST /api/reservations
    5. Backend memvalidasi tidak ada bentrok jadwal
    6. Jika valid, data tersimpan dengan status "pending"
    7. Frontend menampilkan pesan sukses dan reservasi muncul di riwayat
  Alur Alternatif:
    5a. Jika bentrok, backend mengembalikan 409, frontend menampilkan pesan
        error "Ruangan sudah dipesan pada waktu tersebut"

UC-02  MENYETUJUI / MENOLAK RESERVASI
  Aktor       : Admin
  Alur Normal :
    1. Admin login, membuka halaman "Manajemen Reservasi"
    2. Admin melihat daftar reservasi status "pending"
    3. Admin klik "Setujui" → PATCH /api/reservations/:id/approve
    4. Status berubah "approved", notifikasi dibuat untuk user terkait
  Alur Alternatif:
    3a. Admin klik "Tolak" → mengisi alasan → PATCH .../reject
        Status berubah "rejected", notifikasi dibuat berisi alasan

UC-03  MEMBATALKAN RESERVASI
  Aktor       : User (pemilik) atau Admin
  Alur Normal :
    1. User membuka "Riwayat Reservasi", memilih reservasi miliknya
    2. User klik "Batalkan" (hanya muncul jika status pending/approved)
    3. Sistem konfirmasi, lalu PATCH /api/reservations/:id/cancel
    4. Backend memverifikasi req.user.id sama dengan user_id reservasi
       (atau req.user.role === 'admin')
    5. Status berubah "cancelled"
